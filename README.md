# StreamPulse: Media Streaming Observability with Tiger Cloud, TimescaleDB & Grafana

## Project Overview

**StreamPulse** is a time-series observability project built to analyze media streaming performance using **Tiger Cloud**, **TimescaleDB**, **PostgreSQL** and **Grafana**.

The project simulates a real-world streaming platform where users generate playback events and backend services generate API performance metrics. The goal is to identify whether user-facing streaming issues, such as buffering, can be correlated with backend service degradation such as API latency, CPU spikes and errors.

This project was designed as a database troubleshooting and observability case study, focusing on:

- Time-series data modeling with TimescaleDB hypertables
- CSV-based telemetry ingestion into Tiger Cloud
- Incident investigation using SQL
- PostgreSQL query optimization with `EXPLAIN ANALYZE`
- Grafana dashboards for operational monitoring

---

## Business Problem

A media streaming company receives reports that:

> **US Smart TV users are experiencing severe buffering.**

The engineering team needs to answer:

- Is the issue isolated to a specific region or device type?
- Is buffering increasing during a specific time window?
- Is a backend API contributing to the user-facing issue?
- Are API latency, CPU usage and errors increasing at the same time?
- Can database queries used for investigation be optimized?

---

## Solution Summary

I built an end-to-end observability workflow using Tiger Cloud and Grafana.

The workflow includes:

1. Ingesting CSV telemetry data into Tiger Cloud.
2. Modeling playback events and API metrics as TimescaleDB hypertables.
3. Querying time-series data to detect a simulated production incident.
4. Using `EXPLAIN ANALYZE` to inspect query plans.
5. Creating composite and covering indexes to improve investigation query performance.
6. Building a Grafana dashboard to visualize API latency, buffering, CPU usage, request volume, errors and worst-performing segments.

# Database Design

## Core Tables

The project uses three main tables:

- `content_catalog`
- `media_events`
- `api_metrics`

`content_catalog` is a standard PostgreSQL dimension table.

The two high-volume, time-dependent tables were converted into TimescaleDB hypertables:

- `media_events`
- `api_metrics`

## Table Creation

```sql
CREATE TABLE content_catalog (
    content_id      INTEGER PRIMARY KEY,
    title           TEXT NOT NULL,
    content_type    TEXT NOT NULL,
    genre           TEXT NOT NULL,
    release_date    DATE NOT NULL
);

CREATE TABLE media_events (
    event_time      TIMESTAMPTZ NOT NULL,
    event_id        BIGINT GENERATED BY DEFAULT AS IDENTITY,
    user_id         INTEGER NOT NULL,
    content_id      INTEGER NOT NULL REFERENCES content_catalog(content_id),
    event_type      TEXT NOT NULL,
    device_type     TEXT NOT NULL,
    region          TEXT NOT NULL,
    watch_seconds   INTEGER NOT NULL DEFAULT 0,
    bitrate_kbps    INTEGER,
    buffer_ms       INTEGER,
    app_version     TEXT,
    PRIMARY KEY (event_time, event_id)
);

CREATE TABLE api_metrics (
    metric_time     TIMESTAMPTZ NOT NULL,
    service_name    TEXT NOT NULL,
    endpoint        TEXT NOT NULL,
    region          TEXT NOT NULL,
    status_code     INTEGER NOT NULL,
    response_ms     INTEGER NOT NULL,
    request_count   INTEGER NOT NULL,
    cpu_pct         NUMERIC(5,2) NOT NULL
);
```

# Why Hypertables?

A hypertable behaves like a normal PostgreSQL table from the user's perspective, but TimescaleDB automatically partitions its rows into internal chunks based on time.

In this project:

| Hypertable | Time column | Partitioning |
|---|---|---|
| `media_events` | `event_time` | Time-based chunks |
| `api_metrics` | `metric_time` | Time-based chunks |

Columns such as `region`, `device_type`, `endpoint` and `event_type` were **not** used as partition dimensions. They were used for filtering, grouping, Grafana visualizations and index design.

This design supports time-window queries because TimescaleDB can restrict work to the chunks that overlap the requested period.

## Ingestion Challenge: Generated Identity Column

The first upload of `media_events.csv` failed.

The final `media_events` table contained an auto-generated `event_id`, but the CSV did not. The uploader attempted positional mapping, which shifted the remaining values and caused a type mismatch.

Example error:

```text
invalid input syntax for type integer: "pause"
```

## Resolution: Staging Table

I created a staging table whose columns matched the CSV exactly.

```sql
CREATE TABLE media_events_upload (
    event_time      TIMESTAMPTZ NOT NULL,
    user_id         INTEGER NOT NULL,
    content_id      INTEGER NOT NULL,
    event_type      TEXT NOT NULL,
    device_type     TEXT NOT NULL,
    region          TEXT NOT NULL,
    watch_seconds   INTEGER NOT NULL,
    bitrate_kbps    INTEGER,
    buffer_ms       INTEGER,
    app_version     TEXT
);
```

The data was then inserted into the final hypertable using explicit column mapping.

```sql
INSERT INTO media_events (

    user_id,
    content_id,
    event_type,
    device_type,
    region,
    watch_seconds,
    bitrate_kbps,

    app_version

SELECT
    event_time,
    user_id,
    content_id,
    event_type,
    device_type,
```

# Incident Investigation

## Scenario

The simulated incident was designed around this situation:

> US Smart TV users experienced a major buffering spike during the same time window in which the US `/recommendations` API showed elevated latency, CPU utilization and errors.

The investigation was performed in two stages:

1. identify the worst region, device and endpoint combinations;
2. compare the incident window with a baseline period.

---

## Query 1: Buffering by Region and Device

```sql
SELECT
    region,
    device_type,
    COUNT(*) AS buffering_events,
    ROUND(AVG(buffer_ms), 2) AS avg_buffer_ms,
    MAX(buffer_ms) AS worst_buffer_ms
FROM media_events
WHERE event_type = 'buffering'
GROUP BY region, device_type
ORDER BY avg_buffer_ms DESC;
```

## Query 3: US Smart TV Incident Window vs Baseline

```sql
WITH bounds AS (
    SELECT MAX(event_time) AS max_event_time
    FROM media_events
),
labeled_events AS (
    SELECT
        CASE
            WHEN event_time >= max_event_time - INTERVAL '36 hours'
             AND event_time <  max_event_time - INTERVAL '30 hours'
                THEN 'incident_window'
            ELSE 'baseline_other_42_hours'
        END AS time_period,
        event_type,
        buffer_ms
    FROM media_events
    CROSS JOIN bounds
    WHERE region = 'US'
      AND device_type = 'smart_tv'
      AND event_time >= max_event_time - INTERVAL '48 hours'
)
SELECT
    time_period,
    COUNT(*) AS total_events,
    COUNT(*) FILTER (WHERE event_type = 'buffering') AS buffering_events,
    ROUND(
        100.0 * COUNT(*) FILTER (WHERE event_type = 'buffering') / COUNT(*),
        2
    ) AS buffering_rate_pct,
    ROUND(
        AVG(buffer_ms) FILTER (WHERE event_type = 'buffering'),
        2
    ) AS avg_buffer_ms,
    MAX(buffer_ms) FILTER (WHERE event_type = 'buffering') AS worst_buffer_ms
FROM labeled_events
GROUP BY time_period
ORDER BY time_period;
```

### Result Summary

| Time period | Total events | Buffering events | Buffering rate | Average buffer |
|---|---:|---:|---:|---:|
| Baseline | 924 | 63 | 6.82% | 3,130 ms |
| Incident window | 131 | 82 | 62.60% | 10,355 ms |

### Finding

During the incident window:

- buffering rate increased from **6.82% to 62.60%**;
- average buffering increased from approximately **3.1 seconds to 10.4 seconds**.

## Query 4: US Recommendations API Incident Window vs Baseline

```sql
WITH bounds AS (
    SELECT MAX(metric_time) AS max_metric_time
    FROM api_metrics
),
labeled_metrics AS (
    SELECT
        CASE
            WHEN metric_time >= max_metric_time - INTERVAL '36 hours'
             AND metric_time <  max_metric_time - INTERVAL '30 hours'
                THEN 'incident_window'
            ELSE 'baseline_other_42_hours'
        END AS time_period,
        response_ms,
        status_code,
        request_count,
        cpu_pct
    FROM api_metrics
    CROSS JOIN bounds
    WHERE endpoint = '/recommendations'
      AND region = 'US'
      AND metric_time >= max_metric_time - INTERVAL '48 hours'
)
SELECT
    time_period,
    COUNT(*) AS metric_rows,
    ROUND(AVG(response_ms), 2) AS avg_response_ms,
    MAX(response_ms) AS max_response_ms,
    ROUND(AVG(cpu_pct), 2) AS avg_cpu_pct,
    SUM(request_count) AS total_requests,
    COUNT(*) FILTER (WHERE status_code >= 500) AS error_count,
    ROUND(
        100.0 * COUNT(*) FILTER (WHERE status_code >= 500) / COUNT(*),
        2
    ) AS error_rate_pct
FROM labeled_metrics
GROUP BY time_period
ORDER BY time_period;
```

### Result Summary

| Time period | Avg response | Max response | Avg CPU | Error count | Error rate |
|---|---:|---:|---:|---:|---:|
| Baseline | 704 ms | 1,150 ms | 42.48% | 293 | 3.88% |
| Incident window | 2,769 ms | 4,000 ms | 85.05% | 148 | 13.70% |

> The baseline covers 42 hours and the incident covers 6 hours, so the error rate is a more meaningful comparison than the raw error count.

### Finding

During the incident window:

- average API latency increased from **704 ms to 2,769 ms**;
- maximum latency reached **4,000 ms**;
- CPU utilization increased from **42.48% to 85.05%**;
- the normalized error rate increased from approximately **3.88% to 13.70%**.

# Key Findings

The investigation showed that:

- US Smart TV buffering increased sharply during the incident window;
- the US `/recommendations` API showed elevated latency during the same period;
- CPU utilization approximately doubled;
- the normalized API error rate increased;
- request volume remained comparatively stable.

These findings show a strong correlation between backend recommendation-service degradation and user-facing buffering. Because this is synthetic observability data, the project treats this as a **root-cause hypothesis**, not proof of causation.

# Query Optimization

After identifying the incident, I optimized the SQL used for repeated investigation.

The process was:

1. capture the original plan with `EXPLAIN (ANALYZE, BUFFERS)`;
2. inspect scans, filters, rows removed and buffer activity;
3. create an index matching the query access pattern;
4. run `ANALYZE`;
5. rerun the same query and compare results.

---

### First Index Attempt

A composite index on endpoint, region and time was created, but PostgreSQL continued to choose the existing time index.

This was an important learning point:

> Creating an index does not guarantee that PostgreSQL will use it.

The query plan must be checked after every optimization.

### Covering Index

```sql
CREATE INDEX idx_api_metrics_incident_lookup_cover
ON api_metrics (
    endpoint,
    region,
    metric_time DESC
)
INCLUDE (
    response_ms,
    cpu_pct,
    request_count,
    status_code
);

ANALYZE api_metrics;
```

### After Optimization

The new plan used an index-only scan.

Key plan details:

```text
Index Only Scan using idx_api_metrics_incident_lookup_cover
Heap Fetches: 0
Execution Time: 0.633 ms
Buffers: shared hit=4 read=13
```


### Result

| Metric | Before | After |
|---|---:|---:|
| Execution time | 2.888 ms | 0.633 ms |
| Approximate improvement | — | 78% faster |
| Buffers touched | 550 | 17 |
| Rows removed by filter | 15,120 | No post-filter removal shown |
| Scan type | Time index scan + filter | Index-only scan |
| Heap fetches | Table access required | 0 |

---

## Optimization Summary

| Investigation query | Before | After | Improvement |
|---|---:|---:|---:|
| Media buffering | 2.558 ms | 0.359 ms | ~86% |
| API latency | 2.888 ms | 0.633 ms | ~78% |

The dataset is intentionally small, so absolute execution times are already low. The meaningful outcome is the change in plan quality:

- fewer buffers touched;
- fewer irrelevant rows inspected;
- filter conditions moved into index conditions;
- index-only access for the API query;
- zero heap fetches for the optimized API plan.

# Time-Series Aggregation

The Grafana time-series panels use TimescaleDB's `time_bucket()` function.

Example:

```sql
SELECT
    time_bucket('5 minutes', metric_time) AS "time",
    AVG(response_ms) AS avg_response_ms
FROM api_metrics
GROUP BY 1
ORDER BY 1;
```

This converts raw timestamped observations into five-minute monitoring intervals.

Continuous aggregates were also created during the project as a learning exercise, but the final dashboard queries were intentionally kept simple and readable by querying the hypertables directly with `time_bucket()`.

# Challenges and Resolutions

## 1. CSV Column Mapping Failure

**Challenge:**  
The generated `event_id` existed in the final table but not in the CSV, causing positional column mismatch.

**Resolution:**  
Loaded the CSV into a staging table and inserted it into the final hypertable with an explicit column list.

---

## 2. Understanding Hypertable Partitioning

**Challenge:**  
It was initially unclear whether columns other than time were being used to create chunks.

**Resolution:**  
Queried `timescaledb_information.dimensions` and confirmed that each hypertable used only one time partitioning dimension.

---

## 3. Grafana Connection Configuration

**Challenge:**  
Grafana initially attempted to connect to a database named `tsdbadmin`, which was actually the administrative username.

**Resolution:**  
Separated the Tiger Cloud connection string into the correct host, port, database, username, password and SSL fields.

---

## 4. Grafana Macro and Query Mode Issues

**Challenge:**  
The first query was created as a Grafana expression and macros such as `$__timeGroupAlias()` were not parsed correctly in that context.

**Resolution:**  
Switched to a PostgreSQL data-source query in code mode and used TimescaleDB's native `time_bucket()` function.

---

## 5. Table Query Rendered as a Time Series

**Challenge:**  
Categorical result sets produced “data is missing a time field” errors.

**Resolution:**  
Changed both the query format and panel visualization to `Table`.

---

## 6. First API Index Was Not Selected

**Challenge:**  
PostgreSQL continued using the default time index after the first composite API index was created.

**Resolution:**  
Reviewed the new plan and created a covering index that contained both search columns and the columns required by the aggregates. The query then used an index-only scan with zero heap fetches.

- how to design composite and covering indexes around query access patterns;
- how to validate optimization rather than assuming an index worked;
- how to connect Grafana to Tiger Cloud;
- how to use `time_bucket()` for time-series visualizations;

# Final Outcome

StreamPulse demonstrates an end-to-end observability workflow built with Tiger Cloud, TimescaleDB, PostgreSQL and Grafana.

The project:

- ingested more than 279,000 synthetic telemetry records;
- modeled media events and API metrics as hypertables;
- identified a simulated US Smart TV buffering incident;
- correlated the incident with `/recommendations` API degradation;
- improved two investigation queries by approximately 86% and 78%;
- produced a seven-panel Grafana dashboard for operational analysis.
